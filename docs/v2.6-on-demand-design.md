# v2.6 · 真模型按需加载

> 把每一轮 LLM 调用拆成 **"先用工具规划要读什么"** 和 **"基于已读内容流式回答"** 两个 pass。
> Pass 1 是协议契约不是创作；Pass 2 是创作不是规划。
> phase 由 LLM 选，但全程失败信号可观测，**不兜底但全可观测**。

---

## TL;DR

1. 当前 v2.5.4 在 deepseek-chat 上已崩到 MBTI 命中率 3.7%、ok 率 33.75%、报告失败率 66%——继续打磨 L2 静态长 prompt 没有上限可挖。
2. v2.6 的本质改动：常驻 prompt 砍到 ~5KB，所有 phase / technique / domain / example 通过单一工具 `read_skill(names: string[])` 由 LLM 主动按需读取。
3. 工程纪律：**Pass 1 是 tool planning contract**（强制只输出 tool_calls，content 丢弃）；**Pass 2 是流式正文**（不再带工具）；6 项校验全程记录，**不兜底**——任何模型偏离协议都成为 benchmark 信号，不是事故。

## 2. 设计原则

### 2.1 三层职责必须分清

| 层 | 职责 | 写在哪 |
|---|---|---|
| frontmatter `description` | **召回**——让模型知道"这个 skill 可能相关" | 各 `*.md` 顶部 YAML |
| tool `description` | **协议**——告诉模型"怎么拿到内容" | `read_skill` 工具定义 |
| Pass 1 system prompt | **流程**——告诉模型"每轮必须这么做" | `compose_pass1_prompt()` |

三者**叠加生效**，缺一层都会让 LLM 走偏。**绝不混用**——别在 frontmatter 里写流程，别在 tool description 里写抽象价值观（"be thoughtful" 类）。

### 2.2 Agent = Model + Harness

借用 Anthropic 工程博客的定义：harness 是除模型本身之外的所有代码、配置、执行逻辑。v2.6 中 harness 的职责：

- 提供 catalogue（启动时枚举所有 markdown）
- 注入工具（schema 含动态 enum）
- 执行 Pass 1 / Pass 2 双轮
- 协议校验（6 项，记录不纠正）
- 持久化 trace（DB）

**harness 不替模型决策**。它只搭舞台，记录每一步。

### 2.3 不兜底但全可观测

旧设计倾向"模型不听话就由代码兜住"。v2.6 反过来：**模型不听话就让它不听话，但全部记下来**。

理由：
- baseline 已经在 3.7%——没有"产品状态"可保
- 要的不是"看上去还行"，是"知道为什么不行"
- 让兜底逻辑混进生产链路，会让 benchmark 永远看不出真问题

## 3. 架构

### 3.1 总览

```
┌────────────────────────────── 用户消息进来 ─────────────────────────────┐
│                                                                         │
│  Pass 1 · Tool Planning Contract（非流式）                              │
│    system prompt = SKILL body + ETHOS + Runtime State + Skill Index    │
│    tools = [read_skill(names: enum[catalogue])]                        │
│    LLM 必须 call read_skill；message.content 丢弃                       │
│    服务端按 names 取 body，append tool messages                         │
│    loop ≤ 3 轮（让 LLM 第一次少读了可以补一次）                         │
│                                                                         │
│  ──── 协议校验（6 项，不纠正只记录） ────                               │
│                                                                         │
│  Pass 2 · 流式正文                                                      │
│    system prompt = SKILL body + ETHOS + Runtime State + 已加载 skills  │
│    tools = []（明确禁止再调）                                           │
│    stream=True，按 token yield 给 SSE                                   │
│    末尾 STATUS sentinel 解析（仅本 pass 解析）                          │
│                                                                         │
└─────────────────────────── 落库 + yield done ───────────────────────────┘
```

### 3.2 Pass 1 · Tool Planning Contract

**system prompt（Pass 1 专用，与 Pass 2 不同）**：

```
你正在为 OriSelf 这一轮规划 skill 加载。

铁则：
1. 必须调用一次 read_skill(names: string[])
2. names 至少 1 个，最多 8 个
3. names 必须包含 1 个 phase（onboarding / warmup / exploring / midpoint / deep / soft-closing 之一）
4. 当 R1-R3 时，names 必须包含 "exemplary-session"
5. 已经在本会话读过的不要重复
6. 不要在 message content 里写任何对话回复——本轮只做规划

Runtime State: R{round_no} of {target_rounds}
（用户偏好、provider、其他状态）

Skill Index:
- phase-onboarding: R1 开场。TA 已经知道产品是什么…
- phase-warmup: R2 到出现第一条具体场景前…
- phase-exploring: …
- phase-midpoint: …
- phase-deep: 深挖 / 矛盾期…
- phase-soft-closing: …
- reflective-listening: 反映式倾听…
- contradiction-probing: 矛盾探查…
- situational-questions: 场景化提问…
- mbti: 人格维度定义（domain）
- exemplary-session: R1-R3 语气示例
```

**LLM 输出的 `message.content` 整段丢弃**。Pass 1 只信 `tool_calls`。这就是协议本身——不是建议、不是引导，是契约。

### 3.3 Pass 2 · 流式正文

**system prompt（Pass 2 专用）**：

```
SKILL.md body
ETHOS.md
Runtime State

Loaded Skills:
# 本轮阶段指引 · phase-deep
（Pass 1 加载的全文）

# 工具箱 · contradiction-probing
（Pass 1 加载的全文）

# 工具箱 · situational-questions
（Pass 1 加载的全文）

# Domain · mbti（如果 LLM Pass 1 选了它）
…
```

**不带 Skill Index**（已经在 Pass 1 用过）。**不带 tools**（避免模型在正文里又想调）。`stream=True` 直接 yield token。末尾 `STATUS: CONTINUE/CONVERGE/NEED_USER` sentinel 用既有解析器剥除。

预估体量：5-8 KB（vs v2.5.4 的 30+ KB）。

### 3.4 read_skill 工具（执行合同写法）

```python
{
  "type": "function",
  "function": {
    "name": "read_skill",
    "description": (
      "Read one or more named skills before composing the response. "
      "Use this every turn before answering. "
      "Selection order: pick exactly 1 phase based on Runtime State; "
      "then techniques referenced by that phase that genuinely apply this turn; "
      "on R1-R3 also include 'exemplary-session'. "
      "Budget: max 8 names per call. Do not include names already loaded in this session. "
      "Do not invent names. Do not skip this call. "
      "Do not place any reply in message content during this pass."
    ),
    "parameters": {
      "type": "object",
      "properties": {
        "names": {
          "type": "array",
          "items": {
            "type": "string",
            "enum": [<启动时由 skill_loader.list_all_names() 注入>]
          },
          "minItems": 1,
          "maxItems": 8
        }
      },
      "required": ["names"]
    }
  }
}
```

写法纪律：
- **第一行说动作**（"Read one or more named skills…"）
- **明确何时必须用**（"every turn before answering"）
- **明确选择规则**（phase → techniques → R1-R3 加 exemplary）
- **明确预算**（max 8、已读不重复）
- **明确禁止**（不要猜名字、不要跳过、不要写正文）
- **schema 极窄**：`enum` 动态填 catalogue，`minItems=1` `maxItems=8`
- **不写抽象价值观**

### 3.5 协议校验（不纠正只记录）

| 校验项 | 触发条件 | 行为 |
|---|---|---|
| `zero_tool_read` | Pass 1 返回 0 个 tool_calls | 记录，进入 Pass 2 仅靠常驻 prompt 回复 |
| `over_budget` | names > 8 | 记录，截前 8（schema 应该已拦） |
| `invalid_skill` | 名字不在 catalogue | 记录，跳过该项（schema enum 应该已拦） |
| `phase_missing` | names 中 0 个 phase | 记录，不补 |
| `exemplary_skipped` | R1-R3 但 names 中无 exemplary-session | 记录，不补 |
| `redundant_read` | 同 session 重复读 | 记录，返回 `{already_loaded: true}` |

**所有违规都进 benchmark 信号**。这正是要测量的东西。

`Conversation` 表新增字段：

| 字段 | 类型 | 用途 |
|---|---|---|
| `tool_calls_json` | TEXT | Pass 1 LLM 调用清单（原始） |

## 5. KPI · 评测目标

### 5.1 必须打败的 baseline（v2.5.4 deepseek-chat 160 场）

| 指标 | v2.5.4 baseline | v2.6 上线门槛 | v2.6 stretch |

| MBTI 4 字母命中 | 3.7% | > 15% | > 40% |
| 报告 3 次重试失败 | 66% | < 40% | < 10% |

### 5.2 必须新增的过程指标

| 指标 | 含义 | 目标 |
|---|---|---|
| `zero_tool_read_rate` | 多少轮 LLM 完全不读 | < 5% |
| `invalid_skill_rate` | 试图读不存在 name | ≈ 0%（enum 应该清零） |
| `over_budget_rate` | 试图读 > 8 个 | < 1% |
| `phase_mismatch_rate` | LLM 选的 phase 与 Rn 常规对应不一致 | 仅观测，不设阈值 |
| `redundant_read_rate` | 重复读同文件 | < 10% |

**没有过程指标 baseline 就上线 = 不知道下次为什么挂**。

### 5.3 评测复用现有设施

直接用 `benchmark/runner.py`，加一个 `--mode on-demand` flag。160 场标准 fixture 不变。结果目录命名 `results.v2.6.0_ondemand_160`。

## 7. 不做什么（YAGNI）

| 不做 | 理由 |
|---|---|
| **L4 子文件递归读取**（SKILL.md 引用 → 读 → 又引用 → 再读） | catalogue 只允许声明过的文件，禁止隐式 API；v0.2 再考虑 |
| **`parallel_tool_calls=true` 单独开** | `read_skill(names[])` batch tool 已经覆盖"一次读多个"，开 parallel 是冗余复杂度 |
| **多 provider 全打通** | v0.1 先选 deepseek-chat（已有 baseline）打通；qwen / kimi / openai 走同一份 OpenAI compatible 接口，跑一次冒烟即可 |
| **STATUS sentinel 改成结构化字段** | 既有协议保留；只在 Pass 2 解析，避免 Pass 1 tool 消息中误生成被吃掉 |
| **converge 报告轮也改成 tool-use** | converge 是单次长 JSON 输出，本来就不属于"对话轮"；v0.1 不动它 |
| **frontmatter description 全面打磨** | enum + tool description contract 已经够；description 改动留给 v2.6.1 优化 |
| **生产环境兜底逻辑** | benchmark 阶段不兜底（要看真信号）；生产灰度阶段视数据再决定（见第 10 节） |

## 8. ADR · 关键决策记录

> 与 codex CLI 两轮 adversarial 讨论的最终结论。每条都标注谁主张谁让步。

### ADR-1 · phase 选择交给 LLM

- **codex 第一轮主张**：phase 是产品状态机，不能交给 LLM；服务端继续 `choose_phase_key()`，LLM 只选 techniques / examples
- **用户主张**：v2.5.4 在 deepseek 上 33.75% ok / 3.7% MBTI 命中——状态机当前正在产生错的判断，没有"保护"价值
- **codex 第二轮让步**：接受。理由："如果 state machine 当前只贡献 3.7%，继续保护它没有意义"
- **决策**：phase 由 LLM 选，但记录 `chosen_phase_key` + `phase_match_rn` 供分析

### ADR-2 · Pass 1 是协议契约不是创作

- **codex 第二轮主张**：Pass 1 必须强制只输出 tool_calls，message.content 直接丢弃
- **claude 接受**：未自己想到的工程纪律，明显比"软引导"更稳
- **决策**：Pass 1 system prompt 写明"不要在 message content 里写任何对话回复"，服务端无视 content

### ADR-3 · `read_skill(names: string[])` batch vs `name: string` + parallel_tool_calls

- **claude 初稿**：`name: string` 单参 + `parallel_tool_calls=true`
- **codex 主张**：`names: string[]` batch tool 优于 parallel single
- **理由**：一次调用表达完整计划、预算容易校验、去重容易、provider 兼容性更好、日志更清楚
- **决策**：batch tool

### ADR-4 · `names` enum 动态填 catalogue

- **codex 第二轮建议**：schema 加 `enum` 用 catalogue-backed validation
- **用户拍板**：动态填
- **决策**：启动时 `skill_loader.list_all_names()` 注入 schema enum；schema 层面让"试图读不存在名字"不可达

### ADR-5 · Pass 2 不带 Skill Index

- **claude 初稿**：Pass 2 也带完整 Skill Index 以防 LLM"反思应补读"
- **用户拍板**：保留 ETHOS，去掉 Skill Index
- **理由**：Pass 2 是创作不是规划，多 3KB 没价值
- **决策**：Pass 2 = SKILL body + ETHOS + Runtime State + 已加载 skills

### ADR-6 · 完全不读时不兜底

- **codex 第一轮主张**：fallback 到 L2 静态拼装
- **用户拍板**：不兜底
- **codex 第二轮让步**：接受（benchmark 场景），但提醒"生产灰度阶段建议 hard gate"
- **决策**：v0.1 benchmark 不兜底；生产灰度阶段（见第 10 节）观察 `zero_tool_read_rate` 决定是否补 hard gate

### ADR-7 · Tool description 写成执行合同

- **codex 主张**：description 写法是动作 + 必须用 + 顺序 + 预算 + 禁止 + 正例；不写抽象价值观
- **claude 接受**：之前写法过于"营销"
- **决策**：见 §3.4 read_skill 工具定义全文

## 10. FAQ

**Q：为什么不用 Anthropic 官方那种"通用 bash + Read 工具让 Claude 自己 cat SKILL.md"？**
A：那是给 Claude Code 这种通用 agent 沙箱设计的；oriself 是产品后端，文件系统暴露既无必要也增风险。`read_skill(names)` 是同一思想的领域专用版本，安全性更好、表达力够用。

**Q：如果 deepseek 真的不调 read_skill 怎么办？**
A：进 Pass 2 仅靠常驻 prompt（SKILL body + ETHOS + Runtime State）回复。质量肯定降，但记 `zero_tool_read=true` 进 benchmark。这正是要测量的——如果某个 provider 系统性偷懒，benchmark 会立刻显形。**不偷偷补救**。

**Q：多 provider 兼容性怎么保证？**
A：v0.1 只对 deepseek-chat 做端到端验证（已有 baseline）。其他 OpenAI compatible provider（qwen / kimi / openai / 302-gemini）走同一份 `call_tools_only()` 实现，差异仅在 base_url + key。哪个 provider 的 tool-use 不达标，benchmark 数据会指出来——届时再决策是优化还是放弃支持。

**Q：STATUS sentinel 还要吗？会不会被 tool 消息搞乱？**
A：保留。但只在 Pass 2 解析。Pass 1 输出的 content 直接丢弃，永远不进 sentinel 解析器。

**Q：v2.6 后 v2.5.x 的 frontmatter `needs:` 字段还有用吗？**
A：v2.6 不再读 `needs`（因为 phase 由 LLM 选，不再由 Python 拼装时去读 phase 的 needs 列表）。但保留——它是文档化的"phase 与 technique 的常规对应关系"，对人类读者和未来回退仍有价值。

**Q：如果 benchmark 跑出来 v2.6 还是不如 v2.5.0 mimo-pro 怎么办？**
A：那说明问题不在加载机制而在模型能力。v2.6 的成功标准是**打败 v2.5.4 deepseek baseline**（33.75% / 3.7% / 66%），不是打败 v2.5.0 mimo（81.9% / 42.7% / 8%）。后者属于 stretch goal，达不到不影响上线。
